\section{Introducción al estado actual del proyecto}
% =====================================================
El siguiente informe detalla el estado actual del desarrollo del proyecto \textit{SupaClock},
dando continuidad al diseño establecido en el informe de diseño preliminar entregado el 7 de abril de 2026. La motivación, los requerimientos técnicos y el diagrama de bloques de alto nivel descritos en aquel
documento se mantienen vigentes, por lo que en este informe \textbf{no se reproducen}, sino que se
referencian. El presente documento se estructura en torno a cinco ejes fundamentales:
diagrama de bloques de bajo nivel \textit{actualizado}, planificación contrastada con el estado real
del proyecto, análisis de impacto socioeconómico y de estándares aplicables, modelos y simulaciones
que respaldan las decisiones de diseño, set-up final de pruebas y un diseño de pruebas preliminar
(sección que se incorpora en este reporte para complementar la planificación original).
Adicionalmente, en la \textbf{sección de Implementación y resultados} se documentan todos los bloques
que se logró integrar en la etapa actual de desarrollo.

\subsubsection{Resumen ejecutivo del avance.} Actualmente el sistema cuenta con un
\textbf{firmware funcional integrado} que ejecuta \textbf{siete tareas concurrentes} bajo FreeRTOS
(IMU + pasos, PPG con máquina de estados SPOT, Temperatura/Batería, ECG por DMA on-demand,
GUI LVGL de 7 pantallas, flush BLE agregado y monitor de rendimiento), un \textbf{stack BLE NimBLE}
operativo con \textbf{cuatro características GATT} (incluyendo telemetría TLV agregada y canal de
comandos), una implementación del algoritmo \textbf{Pan-Tompkins} en Python+NumPy
\textbf{validada offline} contra trazas reales de ECG (la integración con Firebase Cloud
Functions se reservó para el hito 60\,\%), y un \textbf{esquemático en KiCad} para el
prototipo SuperMini.
Adicionalmente, en esta iteración se implementaron \textbf{modos de energía persistentes}
(SPORT/NORMAL/SAVER con 8 cadencias parametrizadas y persistencia NVS), un sistema de
\textbf{power management dinámico} con DFS y \textit{light sleep} que alcanza un 75\,\% de
tiempo en estado SLEEP, y se encargó la \textbf{migración al Seeed XIAO ESP32-S3} para resolver
limitaciones de memoria y potencia del C3.
Los componentes de Machine Learning (CNN 1D para HAR) y la PCB SMD propia están en fases
tempranas y se concentran en los hitos 60\,\% y 90\,\%. Se inició el desarrollo de la
\textbf{aplicación móvil en Flutter} que actuará como \textit{gateway} entre el dispositivo y
el backend; para la fase de diseño actual, las trazas BLE se validan con el cliente de
escritorio \texttt{supaclock\_monitor.py} (Python + \texttt{bleak}), que grafica biometría
en tiempo real y exporta CSV. La integración \textit{end-to-end} con Firebase
(autenticación, reglas Firestore, despliegue de Cloud Functions) está planificada para el
hito 60\,\%.

\subsubsection{Cambios respecto del diagrama original.} La estructura general del bus es la misma;
las modificaciones son:
\begin{itemize}
    \item \textbf{Bus I2C}: confirmado a 400\,kHz con cuatro esclavos físicos
    (MAX30102 \texttt{0x57}, MAX30205 \texttt{0x48}, BMI160 \texttt{0x68}, MAX17048 \texttt{0x36}),
    accedido bajo el mutex \texttt{xSensorDataMutex} (FreeRTOS) para serializar las cuatro tareas
    concurrentes que comparten el bus.
    \item \textbf{ADC continuo (ECG)}: el AD8232 ya \textit{no} se concibe como un canal ADC
    \textit{single-shot}; la implementación actual usa el periférico \texttt{adc\_continuous}
    del ESP-IDF a 20\,kHz con DMA, decimando por 40 hacia 500\,Hz para BLE. Esto se justifica en
    la \rfigura{fig:rtos}.
    \item \textbf{Botones}: pasaron de tres a \textbf{dos botones} (\texttt{NEXT} GPIO10 y
    \texttt{SELECT} GPIO1) \textit{con short-press y long-press} gestionados por
    \texttt{gpio\_buttons} (debounce SW por \texttt{esp\_timer}); el pin de despertar de
    \textit{deep sleep} se canaliza ahora por \texttt{BTN\_SELECT}.
    \item \textbf{LVGL}: se confirmó el buffer parcial de 14.4\,KB (240$\times$30 px $\times$ 2\,B);
    el GUI quedó dividido en \textbf{siete pantallas} (5 ciclables: Home, Bio, HR\_Spot, ECG, Menú;
    2 sub-pantallas: Modo Energía, Configuración Auto-off) con navegación por estado.
    \item \textbf{Backend Firebase}: el algoritmo Pan-Tompkins (\texttt{firebase/functions/main.py})
    fue \textbf{escrito y validado offline} en este avance, ejecutándolo con
    \texttt{scripts/test\_pt.py} contra los CSV crudos del AD8232. El despliegue como Cloud
    Function (\texttt{firestore\_fn.on\_document\_created}), la autenticación Google y las
    reglas de Firestore se concentrarán en el hito 60\,\%, en paralelo con la primera versión
    del \textit{gateway} móvil en Flutter.
    \item \textbf{Migración a ESP32-S3}: se encargó el módulo \textit{Seeed XIAO ESP32-S3}
    (en tránsito) para resolver las limitaciones de potencia (brown-outs del LDO actual) y de
    memoria (techo SRAM de 320\,KB insuficiente para TinyML). La decisión se justifica
    cuantitativamente en la sección~\ref{sec:modelos}.
    \item \textbf{TinyML}: aún \textit{no} se entrenó el modelo CNN 1D (depende de la
    recolección de datos reales del BMI160 sobre los integrantes); por eso se mantiene como bloque
    pendiente. El contador de pasos \textit{algorítmico} (umbral adaptativo) sí se implementó.
    \item \textbf{PCB SMD propia}: se completó la primera versión del esquemático en KiCad incluyendo la sección de potencia, los footprints
    importados y los buses I2C/SPI. Sin embargo, esta versión \textbf{no será manufacturada}
    dado que la migración al ESP32-S3 requiere un rediseño del módulo MCU; ya se trabaja
    en la versión actualizada para dejar definitivamente la protoboard.
\end{itemize}

\subsection{Carta Gantt corregida}
La carta Gantt presentada en el informe anterior ha sido reemplazada por una versión completamente actualizada. Este cambio responde a la falta de especificidad de la planificación original y a la necesidad de reevaluar los porcentajes de avance (pesos) asignados a cada hito, los cuales se encontraban infravalorados frente a la carga real de trabajo. Esta corrección metodológica permite reflejar con mayor precisión el esfuerzo de ingeniería invertido hasta la fecha y reajusta la importancia de los nuevos hitos por alcanzar.

Los siguientes ajustes estructurales quedan reflejados en la nueva revisión maestra:
\begin{itemize}
    \item \textbf{Reestructuración por Fases Específicas:} Las tareas genéricas del plan original se dividieron en 8 fases de desarrollo secuenciales y lógicas (desde ``Planificación inicial'' hasta ``Cierre técnico''), eliminando la ambigüedad en el alcance de cada módulo.
    
    \item \textbf{Ajuste de Pesos y Avance Real:} Se reasignaron los porcentajes de impacto de cada tarea para reflejar su complejidad real en el sistema. Hitos críticos como el ``Firmware base con FreeRTOS'' (7\,\%) y la ``Integración sensores + MCU + BLE'' (9\,\%) ahora poseen un peso acorde a su dificultad técnica. Con esta recalibración, el avance real consolidado del proyecto se sitúa en un 45\,\%.
    
    \item \textbf{Validación de Subsistemas en Paralelo:} Las tareas de validación de sensores e integración se desglosaron. La validación de la IMU y el algoritmo de pasos se encuentra completa (100\,\%), mientras que los flujos de PPG (HR y SpO\textsubscript{2}), ECG y la integración de la GUI local junto con la comunicación BLE avanzan sólidamente con un 70\,\% de progreso.
    
    \item \textbf{Machine Learning y Datos:} La tarea general de \textit{Machine Learning} se separó explícitamente en ``Recolección de datos reales'' y ``Entrenamiento de modelo'', transparentando que el entrenamiento depende de la primera. Ambas fases se reportan como ``Iniciadas'' (5\,\% de avance) basándose en las sesiones preliminares registradas.
    
    \item \textbf{Desarrollo de PCB Granular:} El diseño de la placa se dividió en hitos técnicos específicos y medibles. El esquemático de la PCB propia se reporta como finalizado (100\,\%), el ruteo y envío a manufactura está en curso (40\,\%), y se planificaron formalmente las etapas futuras de \textit{bring-up} y validación eléctrica tanto para la PCB inicial como para la versión final SMD.
    
    \item \textbf{Visibilidad de Estados:} El nuevo formato incluye una semaforización clara del estado de las tareas (``Listo'', ``Curso'', ``Iniciado'', ``Nada''), confirmando que el desarrollo avanza según los tiempos estipulados para llegar a la integración física.
\end{itemize}

\label{sec:impacto}

\subsection{Aplicaciones e impacto socioeconómico, ambiental y en salud}
\subsubsection{Salud pública.} El proyecto se inserta en la línea de los \textit{wearables} de
fitness, pero con un sesgo hacia el monitoreo \textbf{biométrico de uso continuado}, no de
notificación social. Considerando que sólo el 49,7\,\% de los hombres y el 40,3\,\% de las
mujeres mayores de 18 años en Chile cumplen niveles adecuados de actividad física \cite{encuesta_af, obesidad_chile},
y que el 42\,\% de la población adulta presenta sobrepeso \cite{encuesta_af, obesidad_chile}, un dispositivo que
permita auto-monitorizar pasos, frecuencia cardíaca, SpO\textsubscript{2} y temperatura sin
depender de la suscripción a una plataforma \textit{cloud} de pago tiene impacto directo en
políticas de promoción de la salud cardiovascular y metabólica \cite{minsalwearables}.
Adicionalmente, el modo ECG (toma puntual de la Derivación I, 30\,s) abre la puerta a tamizaje
de fibrilación auricular en atención primaria, en la línea de aplicaciones ya validadas
clínicamente con relojes comerciales \cite{Shin2019}.

\subsubsection{Impacto socioeconómico y de equidad.} El BOM consolidado en el informe anterior
totaliza \$\,58.301\,CLP para la versión prototipo y se proyecta a \$\,88.301\,CLP con la versión
Pro en PCB SMD (4 capas, JLCPCB). Esto contrasta con dispositivos comerciales equivalentes (Apple Watch SE: \$\,250.000
CLP; Galaxy Watch FE: \$\,160.000\,CLP) y con bandas de bajo costo (Mi Band: \$\,30.000\,CLP),
posicionando a SupaClock como una alternativa \textbf{de costo medio con potencial de
fabricación local}, capaz de ser distribuida a través de programas de salud pública o de
investigación universitaria. La filosofía de ``Cero Distracciones'' adoptada en el diseño
busca \textbf{evitar} la carga cognitiva por sobre-notificación descrita en
\cite{Kushlev2016, Pielot2014}, abordando un problema de salud mental relevante que las
pulseras genéricas \textit{exacerban} en lugar de resolver.

\subsubsection{Impacto ambiental.} Se prioriza una arquitectura \textit{Edge}-céntrica que
\textbf{minimiza la transmisión de datos}: la telemetría continua viaja a 1\,Hz (13\,B/s),
el ECG sólo se transmite durante la ventana puntual de 30\,s. Esta decisión reduce el
\textit{footprint} energético del sistema completo (dispositivo + radio + datacenter),
en línea con literatura reciente de \textit{green AI} para wearables. Adicionalmente:
\begin{itemize}
    \item La selección de componentes prioriza encapsulados \textbf{libres de plomo} (RoHS-2,
    Directiva 2011/65/EU \cite{rohs}); todos los IC propuestos cumplen.
    \item Se descartó la tecnología \textit{Memory LCD} de Sharp por su alto costo y la
    \textit{OLED} por su menor durabilidad (burn-in), favoreciendo el IPS LCD que tiene una
    vida útil sobre 50\,000\,h.
    \item La batería LiPo 502030 se selecciona en celdas con BMS integrado y certificación
    IEC\,62133-2 \cite{en62133}, lo que permite un fin de vida controlado y reciclable a
    través de los puntos de acopio chilenos (Chilenter, MUR).
\end{itemize}

\subsubsection{Aspectos éticos.}
\begin{itemize}
    \item \textbf{Privacidad de datos biométricos:} las lecturas de HR, SpO\textsubscript{2}, temperatura
    y ECG son \textit{datos sensibles de salud} bajo el espíritu de la Ley 19.628 (Chile) y la
    GDPR (UE). El backend Firebase \textbf{planificado} para el hito 60\,\% requerirá
    autenticación Google y las reglas de Firestore sólo permitirán acceso al propietario
    (\texttt{request.auth.uid == userId}). Los datos de ECG quedarán asociados al UID del
    usuario y \textbf{nunca al MAC} del dispositivo BLE\@. No se transmitirán datos a
    terceros distintos del usuario que loguea su sesión.
    \item \textbf{Limitaciones declaradas como dispositivo no-médico:} el sistema se diseña
    como \textit{wearable de fitness} con la disclaimer explícita de que las mediciones
    \textbf{no constituyen diagnóstico clínico}. No se persigue homologación FDA / ISP. La
    interfaz indicará un \textit{disclaimer} en la pantalla de ECG.
    \item \textbf{Sesgo en el modelo de Machine Learning:} la futura clasificación HAR (caminar,
    correr, reposo, caída) se entrenará con datos del propio equipo de desarrollo y
    voluntarios; se documentarán las características del dataset (edad, género, IMC) para que
    la generalización sea auditable. Se evitarán técnicas \textit{black-box} sin métrica de
    confianza para la detección de caída.
\end{itemize}

\subsubsection{Riesgos adicionales del LiPo.} Aunque ECMA-287 no aborda específicamente celdas

\subsection{Estándares relacionados con el área del proyecto}
A continuación se enumeran los estándares aplicables y las especificaciones derivadas que
\textit{SupaClock} adopta para asegurar su seguridad e interoperabilidad. Se analizan cuatro
normativas adicionales a la ECMA-287 previamente discutida.

\subsubsection{Bluetooth Core Specification 5.3 \cite{ble_core_spec}.}
\begin{itemize}
    \item \textit{LE 1M PHY} se selecciona como \textit{phy} primario (compatibilidad con
    teléfonos $\geq$ Android 5.0).
    \item Connection Interval = 15\,ms y \textit{slave latency} = 4 para reducir consumo en
    estado idle del dispositivo (gateway).
    \item MTU negociado a 247\,B para minimizar overhead: cada paquete BLE transporta hasta
    244\,B útiles, suficiente para los \textit{chunks} de ECG (10 muestras int16 = 20\,B) o
    los paquetes de telemetría TLV agregada.
    \item \textit{Advertising interval} configurado a \textbf{1000\,ms} (1600 unidades de
    0,625\,ms) para minimizar la actividad de radio en estado no-conectado, habilitando que el
    SoC entre en \textit{light sleep} entre intervalos.
    \item \textit{Modem Sleep BT} habilitado en \texttt{sdkconfig.defaults}: el controlador
    Bluetooth suspende el radio durante los intervalos de conexión, reduciendo significativamente
    el consumo cuando hay un enlace activo pero sin datos pendientes.
    \item \textit{Pairing}: \textit{Just Works} \textbf{sin bonding} (compatibilidad con
    BlueZ/Linux y entornos de testing); las claves no se persisten para facilitar la reconexión
    limpia desde distintos clientes de desarrollo.
    \item Servicios \textbf{0x180A} (Device Information) y \textbf{0x180F} (Battery Service)
    estándar, más un servicio custom de telemetría \textbf{0xFF00} con UUIDs cortos 16-bit
    y 4 características (IMU, TLV agregado, ECG streaming, Comandos RX).
\end{itemize}

\subsubsection{IEEE 11073-10406 (\textit{Personal Health Devices --- Basic ECG}) \cite{ieee11073}.}
Este estándar especifica las propiedades mínimas de un ECG personal monitorizado: rango
$\pm$5\,mV, ruido referido a entrada $<$30\,$\mu V_{pp}$, ancho de banda 0.05--40\,Hz, tasa
de muestreo $\geq$\,250\,Hz para diagnóstico básico. \textit{SupaClock} cumple parcialmente:
el AD8232 entrega 0.5--40\,Hz con ganancia $\sim$1100, la decimación a 500\,Hz supera el
mínimo, pero el modo de derivación es \textit{single-lead} (Lead I bimanual) por lo que el
dispositivo se declara como \textbf{ECG personal de tamizaje}, no de Holter.

\subsubsection{USB Type-C Cable and Connector Specification, Rev.~2.3 \cite{usbpd}.}
El conector USB-C del prototipo opera en modo \textit{legacy 5\,V/500\,mA} (USB 2.0). El
módulo TP4056 ya declara compatibilidad con esta categoría. Para la versión Pro se evalúa
soportar 5\,V/1.5\,A (CC pull-down 22\,k$\Omega$) si se mantiene el TP4056, lo que ahorra
$\sim$\,40\,\% del tiempo de carga.

\subsubsection{IEC 60601-1 (\textit{Medical electrical equipment}) \cite{iec60601}.}
Aunque \textit{SupaClock} no se homologa como dispositivo médico, se adoptan voluntariamente
los criterios de \textbf{aislamiento de paciente tipo BF} para el subsistema ECG: el
front-end AD8232 tiene impedancia de entrada $>$10\,M$\Omega$ y corriente de fuga DC
$<$10\,$\mu$A, dentro del límite BF de 100\,$\mu$A. Esto permite que las versiones futuras
del proyecto puedan ser presentadas a homologación con cambios mínimos.

\subsubsection{RoHS-2 (Directiva 2011/65/EU) \cite{rohs} e IEC 62133-2 \cite{en62133}}
ya descritas en la sección de impacto ambiental.

\subsection{Modelo de autonomía energética}

\subsubsection{Cota superior del consumo medio.} A partir de la fracción de tiempo en cada
estado de potencia (§\,\ref{sec:pm}, $f_{sleep}=0{,}75$), el consumo medio queda acotado por:
\begin{equation}
    \bar{I} \leq f_{active}\cdot I_{active} + f_{sleep}\cdot I_{sleep}
    \label{eq:autonomia}
\end{equation}
que para el modo ECO con pantalla apagada ($I_{active}=17$\,mA, $I_{sleep}\approx 1$\,mA
estimado de la hoja de datos del C3) arroja $\bar{I} \lesssim 5{,}0$\,mA. Con la celda 502030
de 250\,mAh esto proyecta una autonomía nominal $>$\,40\,h en modo ECO, holgadamente sobre
el requerimiento de 12\,h. La cifra debe ratificarse empíricamente sobre la plataforma S3.

\subsection{Validación del algoritmo de pasos}
Se compararon \textbf{ambas variantes} del contador de pasos (la versión C3 en aritmética
entera del Anexo\,A y la versión S3 basada en FFT) sobre la misma traza del BMI160,
simulándolas \textit{offline} en Python (\texttt{scripts/algo\_simulator.py}) con un
pipeline idéntico al embebido. El \textit{ground-truth} se registró manualmente
mientras se grababa la sesión.

    captura del BMI160 (\textit{ground-truth} = 67 pasos). Panel superior: variante C3

\subsubsection{Resultado.} La variante \textbf{C3} reportó 60 pasos vs.\ 67 reales,
\textbf{error de $\sim$10,4\,\%} (sub-conteo, principalmente por pasos de baja amplitud al
inicio que no superan el umbral midpoint hasta que la ventana se actualiza). La variante
\textbf{S3-FFT} reportó 65 pasos, \textbf{error de $\sim$3,0\,\%}, gracias a que la
detección espectral identifica directamente la cadencia rítmica sin depender de la
amplitud absoluta. Pruebas de campo adicionales (no incluidas en el gráfico) arrojaron
errores del mismo orden de magnitud para ambas variantes. La conclusión operativa es que
la migración a la versión FFT del S3 reduce a un tercio el error de conteo y elimina el
\textit{tuning} manual del umbral.

\subsubsection{Limitación detectada y migración planificada.} Durante las pruebas de campo se
identificó que agitar la mano en el aire genera falsos positivos, ya que el algoritmo opera
exclusivamente en el dominio del tiempo y no valida la periodicidad de la señal. Para el
siguiente hito se planifica migrar a un análisis frecuencial basado en FFT (ventana de
3\,s, validación de cadencia rítmica entre 1 y 3\,Hz), aprovechando la librería
\texttt{ESP-DSP} con aceleración SIMD del ESP32-S3.

\subsection{Modelo de concurrencia I2C bajo FreeRTOS}
La preocupación principal al pasar de pruebas individuales a integración fue la potencial
\textit{starvation} en el bus I2C de 400\,kHz, compartido por cuatro tareas con prioridades
distintas (\rfigura{fig:rtos}). Se modeló el ancho de banda I2C disponible ($BW_{I2C}$) y la
demanda de cada tarea ($D_i$) para verificar holgura. Las cadencias corresponden al peor caso
(modo SPORT).

\subsubsection{Cálculo.} Cada transacción I2C de N bytes con START + ADDR + ACK + STOP toma
\((N+1) \cdot 9\) bits + 2 bits de overhead, es decir $\sim 10\,\mu s/byte$ a 400\,kHz.
Las demandas son:
\begin{itemize}
    \item BMI160 a 50\,Hz $\times$ 12\,B = 600\,B/s $\Rightarrow$ 6\,ms/s
    \item MAX30102 a 10\,Hz $\times$ 6\,B/sample $\times$ 5 samples/burst = 300\,B/s $\Rightarrow$ 3\,ms/s
    \item MAX30205 a 1/30\,Hz $\times$ 4\,B = 0,13\,B/s $\Rightarrow$ 0,001\,ms/s
    \item MAX17048 a 1/30\,Hz $\times$ 4\,B = 0,13\,B/s $\Rightarrow$ 0,001\,ms/s
\end{itemize}
Total estimado: \textbf{$\sim$9\,ms/s}, lo que representa un \textbf{0,9\,\%} del bus.
La holgura del 99,1\,\% confirma que el mutex \texttt{xSensorDataMutex} no provoca
\textit{starvation} ni jitter perceptible.

\subsubsection{Validación experimental.} Sobre el firmware integrado se contaron los overflows
del FIFO del MAX30102 durante una sesión de $\sim$4\,h continuas (la métrica más sensible al
jitter de la tarea \texttt{hrm\_task}): 0 overflows. El log filtrado por la línea
\texttt{``MAX30102 FIFO overflows acumulados''} no se imprimió, consistente con el modelo.
La prueba 24\,h se realizará una vez disponible la plataforma S3.

\subsubsection{Impacto de las mitigaciones.} El driver ST7789 fue reescrito eliminando el
\textit{polling} activo durante la espera de DMA SPI: la carga de \texttt{gui\_task} cayó de
47\,\% a $<$1\,\%, liberando la CPU para idle. El refactor del ADC del ECG (handle creado
en \texttt{start\_dma}, destruido en \texttt{stop\_dma}) permite que el lock APB\_MAX se
adquiera \textbf{sólo dentro de la pantalla ECG}, en lugar de retenerlo permanentemente.
Estos dos cambios, combinados con el advertising BLE relajado y el Modem Sleep BT, llevaron
la fracción de SLEEP de 0\,\% a 75\,\%.

\subsection{Análisis comparativo de topología algorítmica frente a la industria}
El desarrollo del firmware y los algoritmos biométricos de \textit{SupaClock} se basó en un extenso análisis de las arquitecturas comerciales (tales como Samsung BioActive \cite{samsung_bioactive}, ecosistemas Apple y algoritmos de Garmin/Fitbit). Dado el contexto de recursos acotados (SRAM y potencia de cómputo limitadas en la familia ESP32) frente a los SoCs dedicados de la industria, el diseño algorítmico se adaptó heurísticamente para lograr mediciones precisas sin un costo computacional prohibitivo.

\textbf{Análisis PPG y mitigación de artefactos cinéticos (Motion Artifacts):}
Mientras que la industria de punta (ej. la arquitectura BioActive de Samsung) implementa redes convolucionales dilatadas \cite{kazemi_cnn} y factorizaciones complejas (SVD + ICA \cite{motion_artifact_svd}) sobre arreglos ópticos multicanal para aislar el ritmo cardíaco del ruido de movimiento durante el trote intenso, \textit{SupaClock} opera con un fotodiodo simple (MAX30102). Para solventar la vulnerabilidad ante el ruido cinético sin hardware dedicado, implementamos un enfoque \textit{ad-hoc} de \textbf{Spot Check con Motion Gating}: la \texttt{imu\_task} computa continuamente la derivada de la aceleración (\textit{jerk}) en la muñeca; si la magnitud inercial excede un umbral empírico, la máquina de estados del PPG invalida inmediatamente la toma de la señal. Esta heurística pasiva asegura que los picos sistólicos detectados en reposo sean de alta fidelidad, requiriendo apenas el 1\,\% de los ciclos de reloj que tomaría un filtro adaptativo multicanal.

\textbf{Electrocardiografía (ECG) y extracción morfológica:}
En dispositivos comerciales, la detección de picos (como la "Detección de Dos Rondas" determinista de Samsung o el solapamiento pasivo de tacogramas de Fitbit \cite{fitbit_afib}) ejecuta promedios geométricos iterativos y algoritmos AMPD a bordo para tamizar asimetrías basales y ruidos electromagnéticos. \textit{SupaClock} evita forzar este procesamiento pesado localmente (lo cual monopolizaría el SoC). En su lugar, explota su topología IoT: el ESP32 decima la señal analógica por hardware (DMA a 500\,Hz) y transmite telemetría cruda. El algoritmo clínico \textbf{Pan-Tompkins} ya fue implementado en Python+NumPy y validado \textbf{offline} en este avance (Sec.\,\ref{sec:modelos}); su despliegue en \textit{Firebase Cloud Functions} con operaciones vectorizadas se planifica para el hito 60\,\%, momento en que el procesamiento será escalable en la nube sin saturar la batería del wearable.

\textbf{Reconocimiento de actividad (HAR) y podometría:}
La detección inercial avanzada en relojes comerciales suele apoyarse en redes recurrentes
o convolucionales sobre arreglos multi-IMU para discernir contextos biomecánicos. Acotados
al ESP32-C3 (sin FPU, single-core), la podometría de \textit{SupaClock} opera actualmente
con un umbral adaptativo \textit{midpoint} en aritmética entera (Anexo A) que, como muestra
la \rfigura{fig:simpasos}, sub-cuenta del orden del 10\,\%. Esta limitación motiva en parte
la migración al \textbf{ESP32-S3}, donde ya se preparó una segunda variante del algoritmo
basada en FFT vía \texttt{esp-dsp} (error $\sim$3\,\% en la misma traza) y existe margen
de SRAM/PSRAM para una eventual CNN 1D. El alcance de SupaClock no apunta a igualar las
métricas de un \textit{flagship} comercial, sino a construir una solución \textbf{abierta y
auditable} con un costo y un consumo de recursos un orden de magnitud por debajo, asumiendo
las pérdidas razonables de precisión que ello implica.

\section{Diseño del set-up final de pruebas del producto integrado}
% =====================================================
\label{sec:setup}
Esta sección describe la arquitectura del \textbf{producto final integrado} que será sometido
a validación, junto con el plan de pruebas mecánicas, eléctricas y de uso real. El set-up se
estructura en tres niveles progresivos: mecánico/integración $\to$ eléctrico/funcional $\to$
uso real. Cada nivel define su banco de pruebas y los criterios de aceptación correspondientes.

% -------------------------------------------------------
\subsection{Arquitectura mecánica del producto final}
\label{sec:arq_mecanica}
% -------------------------------------------------------
El dispositivo final se construye sobre una \textbf{sola PCB de 2 capas} (no un sandwich de
dos PCBs) con la siguiente distribución:
\begin{itemize}
    \item \textbf{Cara superior}: Seeed XIAO ESP32-S3 (en socket 2.54\,mm para la v1 o
    castellated SMD para la v2), módulo AD8232 ECG, MAX17048-STEMMA fuel gauge,
    display ST7789 1.69'' y dos botones táctiles.
    \item \textbf{Cara inferior} (contacto piel): únicamente los sensores MAX30102 (PPG) y
    MAX30205 (temperatura corporal).
\end{itemize}

\subsubsection{Carga y alimentación.}
La carga USB-C se realiza directamente por el conector del Seeed XIAO ESP32-S3, que integra
BMS (\textit{LiPo charger} + protección). No se utiliza módulo TP4056 independiente.

\subsubsection{Batería.}
Celda LiPo reciclada de un smartwatch (dimensiones similares a la del Samsung Galaxy
Watch~4: $\sim$247\,mAh, $\sim$30$\times$25$\times$4\,mm). Las dimensiones finales de la
carcasa se ajustarán al hueco que deje la PCB, no al revés.

\subsubsection{Ventana óptica del MAX30102.}
El encapsulado del sensor ya integra su propio cristal óptico; sólo se requiere un
\textit{cutout} circular en la cara inferior de la carcasa que exponga el sensor sin
obstrucción ni filtros adicionales.

\subsubsection{Acople térmico del MAX30205.}
Un pad térmico (Sil-Pad o equivalente) se interpone entre el sensor y una \textbf{placa de
aluminio} fijada en la mitad inferior de la carcasa. Al apernar la PCB con los tornillos de
cierre, la presión sobre el pad queda autocontenida y la transferencia térmica
carcasa$\leftrightarrow$piel es continua sin depender de la posición del usuario.

\subsubsection{Electrodos ECG.}
Dos pernos \textbf{M3 de acero inoxidable 304} roscados desde el exterior de la carcasa,
con la cabeza redonda orientada hacia la piel. El contacto eléctrico entre los pernos
(fijos a la carcasa) y los pads del PCB se realiza mediante \textbf{pogo pins de 2.5\,mm}
soldados en pad SMD sobre la cara inferior del PCB. Esto permite armar y desarmar la
carcasa sin estresar pads de cobre y posibilita cambiar pernos sin desoldar.
Las señales se conducen a las entradas del in-amp del AD8232 (impedancia
$>$\,10\,M$\Omega$), aisladas eléctricamente del rail GND.

\subsubsection{Carcasa.}
Impresión 3D en PLA, dos mitades, dimensiones objetivo $\sim$50$\times$40$\times$13\,mm
para la iteración v1. Forma \textbf{alargada estilo brazalete} (no caja cuadrada), bordes
redondeados, perfil bajo. Cierre con \textbf{4 tornillos hexagonales M2 o M3} desde la cara
inferior. Lugs estándar de 20\,mm para correa intercambiable (caucho, nylon o silicona).
Ranura lateral para el conector USB-C del XIAO S3.

% -------------------------------------------------------
\subsection{Banco de pruebas mecánicas (caja + integración)}
\label{sec:pruebas_mec}
% -------------------------------------------------------
Cinco pruebas con criterios de aceptación explícitos:

\begin{enumerate}
    \item \textbf{Encaje y cierre}: armar/desarmar la caja 5 veces con los tornillos hex
    M2/M3. Verificar que el PCB no se desplaza, que los botones siguen alineados con sus
    actuadores, que el placa de aluminio aplica presión consistente sobre el MAX30205, y que
    el \textit{cutout} del PPG no presenta luz ambiente filtrándose al sensor.

    \item \textbf{Aislación eléctrica de los electrodos ECG}: con multímetro en modo
    continuidad, verificar resistencia $>$\,10\,M$\Omega$ entre cada perno M3 y los rails
    GND y 3.3\,V del PCB. Los pogo pins deben hacer contacto \textbf{exclusivamente} con los
    pads de entrada del AD8232.

    \item \textbf{Acople térmico (caja cerrada vs.\ integrado desnudo)}: aplicar dedo durante 30\,s
    sobre el placa de aluminio y comparar la curva de temperatura reportada por la unidad
    cerrada vs.\ una sesión equivalente con el integrado desnudo en protoboard. Caracterizar el
    offset y la constante de tiempo del acople carcasa$\leftrightarrow$piel; ajustar firmware
    si es necesario.

    \item \textbf{Drop test}: dejar caer la unidad ensamblada desde 1\,m sobre superficie
    blanda (alfombra), 3 caídas. Criterio: la unidad sigue funcional, sin desconexión BLE
    persistente, y el IMU registra el impacto.

    \item \textbf{Ergonomía y peso}: portar la unidad en el antebrazo durante 1\,h. La forma
    de brazalete alargado debe distribuir el peso sin presión puntual molesta. Reporte de
    comodidad subjetiva en escala 1--5 por cada integrante del equipo.
\end{enumerate}

% -------------------------------------------------------
\subsection{Banco de pruebas eléctricas y funcionales (unidad cerrada)}
\label{sec:pruebas_elec}
% -------------------------------------------------------
Se repiten las pruebas del §\,\ref{sec:pruebas} (unitarias e integración) ahora con la
\textbf{caja cerrada y sobre brazo humano}, contrastando contra las cifras obtenidas en
protoboardoard. La métrica clave es cuánto se degrada cada lectura por el cerramiento.

\begin{enumerate}
    \item \textbf{HR/SpO$_2$ con ventana óptica vs.\ protoboardoard pelado}: aceptar pérdida
    de 1--2\,\% en SpO$_2$ y 1--3\,BPM en HR por la atenuación del cristal integrado del
    MAX30102 dentro de la carcasa. Adicionalmente, se contrastarán las lecturas contra
    un \textbf{Samsung Galaxy Watch~4 Classic} (sensor BioActive, PPG multicanal) portado
    simultáneamente en la muñeca opuesta, registrando HR y SpO$_2$ en paralelo para
    evaluar la correlación entre ambos dispositivos.

    \item \textbf{ECG con pernos M3 + pogo pins vs.\ clips de prueba directos}: verificar que
    la calidad del trazo (R-peak SNR) no cae más de 3\,dB respecto de la sesión con
    electrodos directos sobre pads expuestos. Se realizará además una toma ECG simultánea
    con el \textbf{Galaxy Watch~4 Classic} (Derivación I, 30\,s) para comparar la
    morfología del complejo QRS y el BPM reportado por ambos dispositivos.

    \item \textbf{Temperatura con placa de Al vs.\ contacto directo}: caracterizar el offset
    estático y la constante térmica de la transferencia caja$\leftrightarrow$piel; ajustar
    offset en firmware si la desviación supera $\pm 0.3$\,°C.

    \item \textbf{BLE con caja PLA cerrada}: verificar que el alcance $\geq$\,10\,m no se
    degrada significativamente (el PLA es transparente a 2.4\,GHz).

    \item \textbf{Autonomía empírica (sin INA219)}: nuevo entorno PlatformIO
    \texttt{test\_battery}, clon reducido de \texttt{test\_general}, que mantiene activo el
    firmware en modo SPORT logueando $V_{cell}$ y SOC del MAX17048 cada 30\,s al CSV vía BLE
    hasta que la batería entre en UVP. La curva de descarga resultante se ajusta a un modelo
    lineal/exponencial y de ahí se extrae la autonomía nominal por modo. Esto aprovecha el
    fuel gauge integrado sin necesidad de instrumentación externa.
\end{enumerate}

\subsection{Pruebas de uso (\textit{team testing})}
\label{sec:pruebas_uso}
% -------------------------------------------------------
Los \textbf{tres integrantes del equipo} portan la unidad por 4\,h cada uno en condiciones
cotidianas (oficina, caminata, escalera). La toma de datos \textbf{no requiere PC de forma

    \item Tiempo hasta la primera medición SPOT exitosa.
    \item Cantidad de SPOT abortados por \textit{motion gating}.
    \item Reporte cualitativo de comodidad (escala 1--5).
    \item Pérdidas de enlace BLE durante uso normal.
    \item Datos exportados a CSV para análisis posterior y, eventualmente, dataset de
    entrenamiento HAR.
\end{enumerate}

\subsection{Pruebas de integración}
\begin{enumerate}
    \item \textbf{Concurrencia I2C}: ejecutar \texttt{main\_app} 24\,h y verificar que
    el contador de overflows del MAX30102 se mantiene en cero (validación experimental del
    modelo de la sección\,\ref{sec:modelos}).
    \item \textbf{Coherencia BLE $\to$ CSV}: encender el dispositivo, conectar
    \texttt{supaclock\_monitor.py}, capturar una sesión de 1\,min y verificar que el CSV
    exportado contiene los TLV de HR, SpO\textsubscript{2}, temperatura y batería con
    timestamps coherentes con los \texttt{boot\_ts\_ms} del header agregado.
    \item \textbf{ECG end-to-end (offline)}: presionar SELECT en pantalla ECG, mantener
    contacto bimanual 30\,s, ejecutar \texttt{scripts/test\_pt.py} sobre el CSV resultante y
    contrastar el BPM/HRV calculado con el conteo manual de R-peaks sobre el trazo crudo
    (objetivo: $\leq$\,2\,BPM, $\pm$\,5\,ms HRV).
    \item \textbf{Coherencia BLE $\to$ Firestore} (\textit{hito 60\,\%}): repetir la prueba
    anterior con la integración cloud, verificando que el documento queda con
    \texttt{processingStatus} = \texttt{completed} y \texttt{processedBPM} dentro de
    tolerancia tras el trigger de Cloud Functions.
    \item \textbf{Resiliencia de batería}: simular UVP forzando $V_{cell}$\,$<$\,3.0\,V con la
    fuente de laboratorio, verificar que el dispositivo entra en \textit{deep sleep} y rechaza
    arranque hasta que se reconecte el USB.
    \item \textbf{Power management}: con \texttt{esp\_pm\_dump\_locks}, verificar que fuera de
    la pantalla ECG no existe ningún lock \texttt{NO\_LIGHT\_SLEEP} activo de forma sostenida.
    Validación experimental del modelo §\,\ref{sec:pm}.
    \item \textbf{Persistencia NVS}: cambiar modo de energía y auto-off de pantalla, hacer
    reboot y deep sleep, verificar que ambos valores se mantienen. Borrar NVS
    (\texttt{nvs\_flash\_erase}) y verificar que se aplica el default SPORT.
\end{enumerate}

\subsection{Pruebas funcionales (sistema completo)}
\begin{enumerate}
    \item \textbf{Sesión cotidiana}: portar el dispositivo durante 4\,h (idealmente) en muñeca, registrar
    pasos manualmente y comparar contra \texttt{stepsHW} y \texttt{stepsSw} reportados.
    \item \textbf{Detección de variaciones térmicas}: utilizar una cámara térmica para registrar la
    temperatura de un objeto/superficie calentado progresivamente (y la del ambiente),
    contrastando ambos perfiles térmicos contra la lectura reportada por el MAX30205.
    \item \textbf{Validación métricas de sensores SPO2 y HR}: usar un pulsioxímetro clínico (como referencia) para realizar mediciones en reposo y contrastar con la métrica reportada por el dispositivo.
    \item \textbf{Caída}: dejar caer el dispositivo desde 1\,m sobre superficie blanda
    (objetivo del modelo HAR final; en esta etapa sólo se verifica que el
    dispositivo no se reinicia mecánicamente y que la lectura del IMU registra el impacto).
    \item \textbf{Aceptación visual del UI}: usuarios externos al equipo (idealmente 5) navegan las
    siete pantallas y reportan el tiempo necesario para iniciar una toma de ECG. Objetivo:
    $\leq$\,30\,s sin tutorial. Además, se les encuestará sobre su experiencia de uso.
\end{enumerate}

\subsection{Plan de validación de la PCB propia}
Cuando el ruteo del PCB SuperMini esté completo y manufacturado:
\begin{itemize}
    \item Inspección visual y eléctrica con multímetro: continuidad de pistas críticas (I2C,
    SPI, ADC), cortocircuitos entre pistas 3.3\,V y GND, etc.
    \item Bring-up secuencial de fases (igual que el firmware: \textbf{Fase 1} I2C/sensores,

    \item Comparación A/B contra el prototipo en placa de desarrollo: ejecutar \texttt{main\_app}

    sensores.
\end{itemize}

\subsubsection{Cálculos / decisiones.} La asignación de prioridades sigue la regla \textit{rate
monotonic}: a mayor frecuencia de muestreo, mayor prioridad. \texttt{ecg\_task} (DMA
on-demand) recibe prioridad 7 para no perder muestras DMA;
\texttt{imu\_task} (20--80\,ms según perfil) recibe prioridad 6; \texttt{hrm\_task} (100\,ms polling FIFO) y
\texttt{gui\_task} (33\,ms) reciben prioridad 5; \texttt{ble\_tx\_task} (flush periódico
del buffer TLV agregado) prioridad 4; \texttt{system\_task} (temperatura y batería) prioridad 3;
\texttt{perf\_monitor\_task} (debug) prioridad 2.
Las tareas usan \texttt{vTaskDelayUntil} para mantener la frecuencia exacta a pesar del
jitter del scheduler.

\subsubsection{Brown-out y secuencia de arranque.} Durante la integración, el equipo encontró
un fallo recurrente: el ESP32-C3 entraba en \textit{brown-out reset} al inicializar
simultáneamente la pantalla y el BLE, ambos consumidores agresivos de corriente. Las
mediciones \textbf{previas a las optimizaciones} mostraban transitorios de hasta
$\sim$110\,mA en la batería (cota inferior del peor caso, dada la limitación de ancho
de banda del TX3). Se implementó una \textbf{secuencia de arranque en tres fases} en
\texttt{app\_main} con \texttt{vTaskDelay} entre cada una: Fase 1 (sensores I2C),
Fase 2 (display), Fase 3 (BLE). El \textit{flushing} explícito del FIFO del MAX30102 en
la Fase 3 evita el desbordamiento causado por los $\sim$1,5\,s de latencia entre Fase 1
y el arranque de \texttt{hrm\_task}. Tras las optimizaciones de §\,\ref{sec:pm_impl},
el peak medido cayó a 69\,mA y los brown-outs desaparecieron, aunque se mantiene la
secuencia escalonada como buena práctica.

\subsubsection{Resultados.} El sistema arranca en $\sim$2,0\,s, ejecuta las siete tareas sin
\textit{stack overflow} (verificado con \texttt{uxTaskGetStackHighWaterMark}) y mantiene las
frecuencias planificadas. Gracias al PM dinámico (§\,\ref{sec:pm}), la CPU
pasa $\sim$95\,\% del tiempo en IDLE (\rfigura{fig:cpupie}).

\subsection{Power management dinámico (light sleep)}
\label{sec:pm_impl}
\subsubsection{Descripción.} Se habilitó el PM dinámico de ESP-IDF con la configuración:
\begin{verbatim}
esp_pm_config_esp32c3_t pm = {
    .max_freq_mhz = 160,
    .min_freq_mhz = 10,
    .light_sleep_enable = true
};
\end{verbatim}
Este esquema permite que el SoC baje la frecuencia a 10\,MHz cuando no hay locks activos y
entre en \textit{light sleep} cuando todas las tareas están en \texttt{vTaskDelay}.

\subsubsection{PM locks implementados.} Se identificaron tres fuentes que impiden el sleep:
\begin{enumerate}
    \item \textbf{ECG PM lock} (\texttt{ESP\_PM\_NO\_LIGHT\_SLEEP}, ``ecg''): adquirido por
    \texttt{ecg\_task} al iniciar \texttt{ad8232\_start\_dma()} y liberado al detener el DMA.
    Impide que el reloj APB se reconfigure durante la adquisición continua, evitando los
    artefactos cuadrados observados en el trazo ECG.
    \item \textbf{NimBLE internal}: el stack BLE adquiere automáticamente un lock APB\_MAX
    durante los eventos de radio (advertising, conexión, notify). Configurar advertising a
    1000\,ms y habilitar Modem Sleep BT minimiza el tiempo de retención.
    \item \textbf{SPI DMA} (driver ST7789): el refactor del driver reemplazó el
    \textit{busy-wait} por \texttt{spi\_device\_polling\_end()} no bloqueante, eliminando la
    retención permanente del lock APB que mantenía la CPU despierta.
\end{enumerate}

\subsubsection{Resultado cuantitativo.} Medido con \texttt{esp\_pm\_dump\_locks} en
\texttt{perf\_monitor\_task}: el sistema pasa un 75\,\% del tiempo en SLEEP, 15\,\% en
APB\_MAX y 9\,\% en CPU\_MAX (\rfigura{fig:pmstates}), vs.\ 0\,\%/92\,\%/8\,\% antes de la
optimización.

\subsection{Migración a Seeed XIAO ESP32-S3}
\label{sec:s3}
\subsubsection{Motivación técnica.} La decisión de migrar al ESP32-S3 se fundamenta en cinco
limitaciones cuantificadas del ESP32-C3 SuperMini actual:
\begin{enumerate}
    \item \textbf{Brown-outs}: el LDO ME6211 no soportaba los transitorios pre-optimización
    ($\sim$110\,mA medidos) del radio BLE arrancando con la pantalla SPI\@. Aunque las
    optimizaciones de software bajaron el peak a 69\,mA y eliminaron el síntoma, el LDO
    sigue siendo el eslabón débil de la cadena de potencia. El Seeed XIAO ESP32-S3 integra
    un regulador buck (SGM6029, $\sim$800\,mA) que da margen amplio para futuras adiciones.
    \item \textbf{SRAM}: el C3 dispone de 320\,KB de SRAM interna. Tras NimBLE, LVGL, buffers
    DMA y los stacks FreeRTOS de las siete tareas, el heap libre estabilizado en
    \texttt{test\_general} es $\sim$99\,KB. El modelo CNN 1D y sus buffers de inferencia
    (\textit{scratch}) requieren entre 20 y 100\,KB según topología, dejando un margen muy
    ajustado para crecer la GUI o la lógica de aplicación.
    \item \textbf{PSRAM}: el S3 ofrece 8\,MB de PSRAM accesible vía OPI, permitiendo alojar el
    modelo TFLite y los buffers de inferencia en memoria externa.
    \item \textbf{Flash}: 8\,MB (vs.\ 4\,MB del C3 SuperMini), margen para OTA y NVS extendido.
    \item \textbf{SIMD/DSP}: el S3 incluye instrucciones vectoriales que aceleran la librería
    ESP-DSP, beneficiando tanto la FFT planificada para el algoritmo de pasos como la
    inferencia del modelo CNN.
\end{enumerate}

\subsubsection{Estado.} Módulo Seeed XIAO ESP32-S3 encargado (en tránsito). Plan de bring-up:
(1) validar compilación y drivers I2C/SPI sin cambios, (2) habilitar PSRAM y mover buffers
grandes, (3) desplegar modelo CNN 1D, (4) validación A/B de consumo contra el prototipo C3.

\subsection{Bloque de Sensores: BMI160 + algoritmo de pasos}
\subsubsection{Descripción.} Driver propio en C (\texttt{lib/bmi160\_driver}) que reemplaza la
API monolítica oficial de Bosch (641\,KB). Implementa: soft-reset, configuración de PMU
(accel + gyro a Normal), ODR 50\,Hz, rango $\pm 2$\,g / $\pm 250$°/s, lectura combinada de
6 ejes, habilitación del step counter de hardware (registros \texttt{0x7A/0x7B}) y reset.

\subsubsection{Algoritmo de pasos (versión C3).} Implementado en \texttt{lib/step\_algorithm}
con dos variantes seleccionadas en compilación según \texttt{CONFIG\_IDF\_TARGET}: una en
aritmética entera para el C3 (sin FPU) y otra basada en FFT para el S3 (con \texttt{esp-dsp}).
La versión C3 opera así:
\begin{enumerate}
    \item Magnitud lineal $|a| = \sqrt{a_x^2 + a_y^2 + a_z^2}$ vía \texttt{int\_sqrt} (Newton entero).
    \item Filtro pasa-bajo IIR de 1\textsuperscript{er} orden con $\alpha=1/4$:
    $\hat{m}_n = (3\hat{m}_{n-1} + m_n)/4$.
    \item Umbral adaptativo midpoint sobre ventana de 50 muestras (1\,s a 50\,Hz):
    $thr = \min + (\max-\min)/2$, sólo si la diferencia pico-a-pico cae en
    $[1500,\,30000]$\,LSB; fuera de ese rango el umbral se eleva artificialmente para
    no gatillar con ruido o movimiento extremo.
    \item Detección por cruce ascendente del umbral con refractario en
    $[300,\,2000]$\,ms entre pasos consecutivos.
    \item \textit{Gating} rotacional: el paso sólo se valida si la magnitud máxima
    del giroscopio en la ventana excede 400\,LSB ($\sim$25\,°/s), evitando falsos
    positivos por vibración traslacional (vehículos, tecleo).
    \item Histeresis temporal: se requieren \texttt{VALID\_STEPS\_THRESHOLD}\,=\,4
    pasos consecutivos para iniciar el conteo, y luego cada paso adicional cuenta
    individualmente.
\end{enumerate}
La versión \textbf{S3} reemplaza la lógica temporal por una FFT radix-2 de 128 puntos
con ventana de Hann, busca el pico espectral en la banda \mbox{$[0{,}78,\,2{,}73]$\,Hz}
(bins 2--7 a $f_s$\,=\,50\,Hz) y deriva el número de pasos como
$N_{pasos}=f_{dom}\cdot 2{,}56\,\text{s}$ por ventana. El código completo está versionado
en el repositorio (Anexo\,A reproduce la versión C3, que es la operativa hoy).
La \rfigura{fig:mlpipeline} muestra los dos pipelines complementarios (contador algorítmico
y futuro clasificador HAR), enfatizando que ambos consumen el mismo flujo del BMI160.

\begin{figure}[H]
    \centering
    \resizebox{0.95\linewidth}{!}{\input{fig_pipeline_ml.tex}}
    \caption{Dos pipelines de procesamiento del BMI160: arriba el contador de pasos
    (implementado), abajo la futura clasificación HAR vía CNN 1D (en desarrollo).}
    \label{fig:mlpipeline}
\end{figure}

\subsubsection{Resultados.} Lectura estable de los 6 ejes a 50\,Hz desde \texttt{imu\_task}.
El test \texttt{test\_bmi160} reporta el conteo cada segundo y permite validar el algoritmo
en frío. Las trazas representativas se procesan \textit{offline} con
\texttt{algo\_simulator.py} contra el \textit{ground-truth} manual (\rfigura{fig:simpasos}):
la variante C3 actual incurre en errores del orden del 10\,\% por sub-conteo en cadencias
lentas, y la variante FFT del S3 (preparada para activarse con la migración) reduce el
error a $\sim$3\,\%.

\subsection{Bloque de Sensores: MAX30102 (PPG)}
\subsubsection{Descripción.} Driver propio en C (\texttt{lib/max30102\_driver}) que implementa:
configuración del modo SpO\textsubscript{2} (RED + IR), pulse width 411\,$\mu$s, sample
average $N=4$, FIFO de 32 muestras, lectura por ráfagas y dos algoritmos: cálculo de HR por
detección de picos sobre la señal IR y cálculo de SpO\textsubscript{2} por la razón de los
ratios AC/DC (R = $\frac{AC_{red}/DC_{red}}{AC_{ir}/DC_{ir}}$). Para evitar overflows del
FIFO, la \texttt{hrm\_task} hace \textit{flush} al arrancar y polling agresivo cada 100\,ms.

\subsubsection{Máquina de estados SPOT.} Se implementó una SM para mediciones puntuales
de HR/SpO\textsubscript{2} con los estados:
\begin{enumerate}
    \item \textbf{IDLE}: sensor en SHDN (modos NORMAL/SAVER entre mediciones).
    \item \textbf{SETTLING}: sensor despertado, se descartan las primeras muestras mientras
    la señal se estabiliza (\texttt{SPOT\_SETTLE\_MS} = 5\,s).
    \item \textbf{MEASURING}: acumulación de muestras para calcular HR y SpO\textsubscript{2}
    con indicador de progreso.
    \item \textbf{DONE}: resultado válido (BPM, SpO\textsubscript{2}, calidad, duración).
    \item \textbf{FAILED}: señal inutilizable (dedo mal posicionado o movimiento excesivo).
    \item \textbf{ABORTED}: cancelación manual por el usuario (botón SELECT).
\end{enumerate}
En modos NORMAL y SAVER, la \texttt{hrm\_task} dispara mediciones SPOT automáticas con la
periodicidad definida por el perfil (\texttt{hrm\_auto\_period\_ms}), apagando el sensor
entre mediciones (\texttt{hrm\_shdn\_between = true}).

\subsubsection{Motion gating.} La \texttt{imu\_task} calcula el \textit{jerk} (derivada
discreta de la aceleración: $|\Delta a|^2 = \Delta a_x^2 + \Delta a_y^2 + \Delta a_z^2$)
y lo informa al driver MAX30102 vía \texttt{max30102\_set\_motion\_level()}. Valores de jerk
por encima de un umbral (80) invalidan la medición SPOT, evitando publicar lecturas ruidosas.

\subsubsection{Resultados.} Sobre el dedo del integrante, el sensor entrega lecturas estables
de 70--75\,BPM y 96--98\,\% de SpO\textsubscript{2}, coincidiendo con el oxímetro Beurer
PO 30 ($\pm$\,2\,BPM, $\pm$\,1\,\% SpO\textsubscript{2}). La latencia hasta convergencia es
$\sim$\,4\,s tras detección de dedo (\texttt{max30102\_finger\_present}).

\subsection{Bloque de Sensores: AD8232 (ECG) con ADC continuo + DMA}
\subsubsection{Descripción.} Implementación completa en \texttt{lib/ad8232\_driver} que reemplaza
el patrón de muestreo \texttt{adc1\_get\_raw()} \textit{single-shot} (incompatible con
50--500\,Hz por el jitter de la tarea) con el periférico \texttt{adc\_continuous} del
ESP-IDF v5.x. Configuración: ADC1 ch.\,0 (GPIO0), 20\,kHz HW, frame DMA de 256\,B, callback
\texttt{on\_conv\_done} que notifica a \texttt{ecg\_task} vía
\texttt{vTaskNotifyGiveFromISR}. La tarea decima por 40 promediando ráfagas de 40 muestras
para entregar 500\,Hz efectivos al BLE, agrupados en chunks de 10 muestras int16
(20\,B/notify) por la característica \texttt{0xFF03}.

\subsubsection{Refactor DMA on-demand.} Para minimizar el impacto del ADC continuo sobre el
power management, el handle del ADC se crea/destruye dinámicamente:
\begin{itemize}
    \item \texttt{ad8232\_start\_dma()}: crea el handle \texttt{adc\_continuous}, inicia la
    captura y adquiere el PM lock \texttt{ESP\_PM\_NO\_LIGHT\_SLEEP} (``ecg'').
    \item \texttt{ad8232\_stop\_dma()}: detiene y destruye el handle, libera el PM lock.
\end{itemize}
Esto garantiza que el lock APB\_MAX sólo se retiene durante la grabación ECG activa,
permitiendo \textit{light sleep} el resto del tiempo. El entorno \texttt{test\_ecg\_raw}
permite capturar ECG sin promedio ni PM para depurar artefactos a nivel de hardware.

\subsubsection{Cálculos.} La elección de 20\,kHz ($\to$ 500\,Hz) viene de dos restricciones:
\begin{enumerate}
    \item El ADC continuo del ESP32-C3 requiere $f_s \geq 20\,kHz$ para operar en modo DMA
    estable (datasheet ESP32-C3 \cite{esp32c3_ds}).
    \item El estándar IEEE 11073-10406 \cite{ieee11073} fija $f_s \geq 250\,Hz$ para ECG
    personal de tamizaje. 500\,Hz dobla este mínimo.
\end{enumerate}

\subsubsection{Resultados.} Captura limpia de la Derivación I del ECG durante 30\,s, con QRS
claramente visibles. El archivo \texttt{supaclock\_ecg\_20260422\_194732.csv} contiene una
captura representativa que se procesa correctamente en la corrida offline de Pan-Tompkins
(BPM = 76, HRV = 32,4\,ms; ver Sec.\,\ref{sec:modelos}).

\subsection{Bloque de Sensores: MAX30205 (Temperatura)}
\subsubsection{Descripción.} Driver mínimo (\texttt{lib/max30205\_driver}, $\sim$50 líneas) que
inicializa el sensor en modo \textit{continuous} y entrega la temperatura en \textit{float}
combinando el registro \texttt{0x00} (16-bit, resolución 0.00390625\,°C/LSB).

\subsubsection{Resultados.} Lectura estable a 1\,Hz desde \texttt{sensor\_task}, valores
$\sim$28--32\,°C en reposo (temperatura de la piel del antebrazo), coincidiendo con un
termómetro IR comercial dentro de $\pm$\,0.3\,°C. Variación dinámica al colocar el dedo
sobre el chip: incremento de $\sim$2\,°C en 5\,s.

\subsection{Bloque de Energía: BMS + LDO + Fuel Gauge}
\subsubsection{Descripción.} Compuesto por tres módulos físicos: TP4056+DW01A (módulo Aliexpress
con USB-C), LDO ME6211 SMD soldado a placa puente, y MAX17048 importado del PCB Adafruit
STEMMA (clonado a la librería \texttt{SupaclockaLIB.kicad\_sym}). El driver
\texttt{max17048\_driver} expone \texttt{max17048\_get\_voltage()} y \texttt{max17048\_get\_soc()}
sobre el bus I2C compartido.

\subsubsection{Validación eléctrica.} Con la fuente DC variando entre 3.0\,V y 4.3\,V, la salida del
LDO se mantiene en 3.30\,V $\pm$\,5\,mV hasta una caída de batería al 5\,\%
($V_{cell}\approx$\,3.05\,V) cuando empieza a colapsar (esperado por dropout). El BMS corta
la entrada al detectar 4.25\,V (sobre-voltaje) y a $V_{cell}\leq$\,2.55\,V (sub-voltaje).

\subsubsection{Resultados.} El driver MAX17048 entrega $V_{cell}$ y SOC estables a 1\,Hz, con
detección automática de POR y \textit{Quick Start} en frío. La validación cuantitativa de
autonomía (descarga continua de la celda 502030 instrumentada con coulomb counter) se
realizará sobre la plataforma definitiva ESP32-S3, en línea con el disclaimer instrumental
de la sección\,\ref{sec:autonomia}.

\subsection{Bloque de Comunicaciones: BLE NimBLE}
\subsubsection{Descripción.} Reescritura completa del stack BLE migrando de Bluedroid (cuyo
\textit{footprint} de 270\,KB excedía la SRAM disponible) a \textbf{NimBLE} (host: $\sim$\,90\,KB).
Implementado en \texttt{lib/ble\_telemetry} con \textbf{cuatro características GATT} bajo el
servicio custom \texttt{0xFF00}:
\begin{itemize}
    \item \texttt{0xFF01} -- Telemetría IMU cruda (12\,B/notify, cadencia según perfil, desactivable desde menú).
    \item \texttt{0xFF02} -- \textbf{Telemetría TLV agregada}: buffer de hasta 200\,B con
    header (\texttt{ble\_agg\_header\_t}: \texttt{boot\_ts\_ms} u32, \texttt{power\_mode} u8,
    \texttt{payload\_len} u8) seguido de records TLV (tipo u8, largo u8, datos). Siete tipos
    definidos: HR, SpO\textsubscript{2}, Temp, Bat, Steps, Spot Result, Mode Event. Flush
    periódico por \texttt{ble\_tx\_task} según cadencia del perfil (1--60\,s).
    \item \texttt{0xFF03} -- ECG streaming (chunks de 20\,B a 500\,Hz efectivos).
    \item \texttt{0xFF04} -- \textbf{Comandos RX} (WRITE, no ACK): byte 0x01 = iniciar ECG,
    0x00 = detener ECG.
\end{itemize}
Adicionalmente se incluye el servicio estándar \texttt{0x180A} (Device Information) con
\textit{Manufacturer Name} y \textit{Model Number}. \textit{Pairing}: \textit{Just Works}
\textbf{sin bonding} (compatibilidad BlueZ/tests).

\subsubsection{Optimizaciones de consumo.}
\begin{itemize}
    \item \textit{Advertising interval} = 1000\,ms (1600 $\times$ 0,625\,ms) para permitir
    \textit{light sleep} entre intervalos.
    \item \textit{Modem Sleep BT} habilitado en \texttt{sdkconfig.defaults}.
    \item Nombre del dispositivo: \texttt{``SupaClock\_BLE''}, appearance: \texttt{0x00C1} (Watch).
\end{itemize}

\subsubsection{Resultados.} Conexión estable con \texttt{supaclock\_monitor.py} (Python +
\texttt{bleak}). Con la potencia de transmisión por defecto de NimBLE no se observa pérdida
de enlace en distancias del orden de 10\,m en interior (sin medición formal de RSSI ni
latencia, pendiente para la fase de validación instrumentada). Sin desconexiones inesperadas
durante las sesiones de prueba.

\subsection{Bloque de Interfaz Local: GUI multi-pantalla LVGL}
\subsubsection{Descripción.} \textbf{Siete pantallas} construidas con LVGL 8.4 sobre el ST7789
a 30\,FPS, gobernadas por \texttt{gui\_task}:
\begin{enumerate}
    \item \texttt{SCREEN\_HOME}: reloj, pasos, batería (arco), HR, nivel de actividad, modo activo.
    \item \texttt{SCREEN\_BIO}: HR, SpO\textsubscript{2}, temperatura, estado del sensor, edad de
    cada medida.
    \item \texttt{SCREEN\_HRSPOT}: interfaz de medición puntual HR/SpO\textsubscript{2} con
    barra de progreso y resultado con calidad.
    \item \texttt{SCREEN\_ECG}: instrucciones, cronómetro de grabación, indicador REC.
    \item \texttt{SCREEN\_MENU}: lista de 7 opciones navegables.
    \item \texttt{SCREEN\_MODE} (sub): selector de SPORT/NORMAL/SAVER con indicador activo.
    \item \texttt{SCREEN\_SETTINGS} (sub): ajuste de auto-off por modo (valores cíclicos
    5/8/15/30/60/120\,s).
\end{enumerate}
La navegación se realiza con dos botones físicos: \textbf{NEXT} (cicla pantallas 1--5 o ítems
en listas; long-press = atrás) y \textbf{SELECT} (acción contextual; long-press = Home).
Cada pantalla tiene su \textit{updater} dedicado que toma snapshot del struct
\texttt{shared\_sensor\_data\_t} bajo mutex y refresca los \texttt{lv\_label\_t} relevantes
sin re-construir widgets.

\subsubsection{Menú principal (7 ítems).}
\begin{enumerate}
    \item Modo Energía (abre sub-pantalla).
    \item Auto-off Pantalla (abre sub-pantalla).
    \item Reiniciar Pasos.
    \item Vincular BLE.

\end{enumerate}

\subsubsection{Backlight PWM y auto-off.} La pantalla usa PWM para controlar el brillo.
Tras un período de inactividad (definido por el perfil activo vía
\texttt{power\_get\_display\_off\_s()}), \texttt{gui\_task} apaga el backlight
(\texttt{st7789\_set\_brightness(0)}) y baja el frame-rate a 10\,Hz, reduciendo la carga
de CPU y habilitando \textit{light sleep} más prolongado. Cualquier pulsación de botón
restaura el brillo al 100\,\% antes de procesar la acción.

\subsubsection{Resultados.} Latencia de transición de pantalla $<$\,50\,ms; sin tearing visible.
El driver ST7789 reescrito eliminó el \textit{busy-wait} durante DMA SPI, reduciendo
la carga de \texttt{gui\_task} de 47\,\% a $<$1\,\% (\rfigura{fig:cpupie}).

\subsection{Bloque de Procesamiento: Pan-Tompkins en NumPy (validado offline)}
\subsubsection{Descripción.} Implementación en Python 3.11 + NumPy de las cinco etapas del

\subsection*{Anexo A: Algoritmo de pasos para C3 (umbral midpoint + gating rotacional)}
\begin{lstlisting}[language=C, caption={Detección de pasos en aritmética entera para ESP32-C3
(\texttt{lib/step\_algorithm/step\_algorithm.c}, extracto de la rama \texttt{\#else}).}, label={lst:steps}]
#define WINDOW_SIZE           50      /* 1 s a 50 Hz */
#define STEP_MIN_TIME_MS      300
#define STEP_MAX_TIME_MS      2000
#define VALID_STEPS_THRESHOLD 4

uint8_t step_algo_update(step_algo_state_t *st,
                         int16_t ax, int16_t ay, int16_t az,
                         int16_t gx, int16_t gy, int16_t gz,
                         uint32_t now_ms) {
    uint8_t new_steps = 0;

    /* 1. Magnitud lineal entera (int_sqrt evita la FPU ausente del C3) */
    uint32_t sum_sq = (uint32_t)((int32_t)ax*ax) +
                      (uint32_t)((int32_t)ay*ay) +
                      (uint32_t)((int32_t)az*az);
    uint32_t mag = int_sqrt(sum_sq);

    /* 2. LPF IIR de 1er orden, alfa = 1/4 */
    st->prev_filtered_mag_sq = st->filtered_mag_sq;
    st->filtered_mag_sq = (st->filtered_mag_sq * 3 + mag) / 4;
    uint32_t cur = st->filtered_mag_sq;

    /* 3. Energía rotacional máxima dentro de la ventana */
    uint32_t gyro_mag = int_sqrt((uint32_t)((int32_t)gx*gx) +
                                  (uint32_t)((int32_t)gy*gy) +
                                  (uint32_t)((int32_t)gz*gz));
    if (gyro_mag > st->max_gyro_val) st->max_gyro_val = gyro_mag;

    /* 4. Min/max dinamico -> umbral midpoint cada WINDOW_SIZE muestras */
    if (cur > st->max_val) st->max_val = cur;
    if (cur < st->min_val) st->min_val = cur;
    if (++st->sample_count >= WINDOW_SIZE) {
        uint32_t diff = st->max_val - st->min_val;
        if (diff > 1500 && diff < 30000)
            st->threshold = st->min_val + diff/2;
        else
            st->threshold = st->min_val + 4000;  /* movimiento muy débil */
        st->sample_count = 0;
    }

    /* 5. Cruce ascendente + refractario + gating rotacional */
    if (st->prev_filtered_mag_sq < st->threshold && cur >= st->threshold) {
        uint32_t dt = now_ms - st->last_step_time_ms;
        if (st->max_gyro_val > 400) {  /* ~25 deg/s, FSR = +/-250 deg/s */
            if (dt >= STEP_MIN_TIME_MS && dt <= STEP_MAX_TIME_MS) {
                st->consecutive_steps++;
                st->last_step_time_ms = now_ms;
                if (st->consecutive_steps == VALID_STEPS_THRESHOLD)
                    new_steps = VALID_STEPS_THRESHOLD;  /* primer commit */
                else if (st->consecutive_steps > VALID_STEPS_THRESHOLD)
                    new_steps = 1;
            } else if (dt > STEP_MAX_TIME_MS) {
                st->consecutive_steps = 1;
                st->last_step_time_ms = now_ms;
            }
        }
        st->max_gyro_val = 0;
    }
    return new_steps;
}
\end{lstlisting}

\subsection*{Anexo C: Pipeline ECG con ADC continuo (DMA) y PM lock on-demand}
\begin{lstlisting}[language=C, caption={Tarea de ECG: DMA on-demand $\to$ down-sample $\to$
BLE notify (\texttt{src/tests/test\_general.c}, función \texttt{ecg\_task}, extracto).},
label={lst:ecg}]
#define ECG_DOWNSAMPLE_RATIO 40    /* 20 kHz HW -> 500 Hz */
#define ECG_BLE_CHUNK_SIZE   10    /* 10 x int16 = 20 B/notify */

void ecg_task(void *pvParameter) {
    uint8_t  dma_buf[AD8232_READ_LEN];
    uint32_t ret_num = 0;
    int16_t  ble_chunk[ECG_BLE_CHUNK_SIZE];
    int      chunk_idx = 0;
    uint32_t sum = 0; int count = 0;
    bool     is_dma_running = false;

    while (1) {
        if (!ble_telemetry_is_ecg_mode_active()) {
            if (is_dma_running) {
                ad8232_stop_dma();
                is_dma_running = false;
#if CONFIG_PM_ENABLE
                if (s_ecg_pm_lock) esp_pm_lock_release(s_ecg_pm_lock);
#endif
            }
            vTaskDelay(pdMS_TO_TICKS(500));
            chunk_idx = 0; sum = 0; count = 0;
            continue;
        } else if (!is_dma_running) {
#if CONFIG_PM_ENABLE
            if (s_ecg_pm_lock) esp_pm_lock_acquire(s_ecg_pm_lock);
#endif
            ad8232_start_dma();
            is_dma_running = true;
        }

        esp_err_t ret = adc_continuous_read(ad8232_get_adc_handle(), dma_buf,
                                            AD8232_READ_LEN, &ret_num,
                                            pdMS_TO_TICKS(100));
        if (ret == ESP_OK) {
            for (int i = 0; i < ret_num; i += sizeof(adc_digi_output_data_t)) {
                adc_digi_output_data_t *p = (adc_digi_output_data_t*)&dma_buf[i];
                uint16_t raw_val = p->type2.data;
                sum += raw_val; count++;
                if (count >= ECG_DOWNSAMPLE_RATIO) {
                    ble_chunk[chunk_idx++] = (int16_t)(sum / ECG_DOWNSAMPLE_RATIO);
                    sum = 0; count = 0;
                    if (chunk_idx >= ECG_BLE_CHUNK_SIZE) {
                        ble_telemetry_send_ecg(ble_chunk, sizeof(ble_chunk));
                        chunk_idx = 0;
                    }
                }
            }
        }
    }
}
\end{lstlisting}

\subsection*{Anexo D: Pan-Tompkins en Python+NumPy (validado offline)}
\begin{lstlisting}[language=Python, caption={Detección adaptativa de R-peaks en NumPy
(\texttt{firebase/functions/main.py}, función \texttt{detect\_r\_peaks}; aún no desplegada
como Cloud Function).}, label={lst:pt}]
def detect_r_peaks(integrated: np.ndarray,
                   original:   np.ndarray,
                   fs: float = 100.0) -> list[int]:
    spki = np.max(integrated[:int(2 * fs)]) * 0.25   # Signal peak
    npki = np.mean(integrated[:int(2 * fs)]) * 0.5   # Noise peak
    threshold1 = npki + 0.25 * (spki - npki)

    r_peaks = []
    refractory_period = int(0.2 * fs)                # 200 ms
    last_peak = -refractory_period

    for i in range(1, len(integrated) - 1):
        if (integrated[i] > integrated[i-1]
                and integrated[i] > integrated[i+1]):
            if integrated[i] > threshold1 and (i - last_peak) > refractory_period:
                spki = 0.125 * integrated[i] + 0.875 * spki
                r_peaks.append(i)
                last_peak = i
            else:
                npki = 0.125 * integrated[i] + 0.875 * npki
            threshold1 = npki + 0.25 * (spki - npki)
    return r_peaks
\end{lstlisting}
